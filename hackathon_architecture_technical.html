SMS via Twilio

User texts the Hey804 number. Twilio webhook hits /api/sms/incoming. Engine returns plain text capped for SMS readability. Twilio wraps in TwiML and delivers.

Web Application

Card-based React UI. Calls /api/chat with channel="web". Returns rich JSON with action_steps, sources, deadlines, handoff options.

Embeddable Widget

One <script> tag. Any partner site gets a compact Hey804 panel. Same engine, same responses, styled for embed.

QR Code Posters

Printed posters at libraries, bus stops. URL includes ?location=southside. Engine returns location-aware greeting and contextual help.

Inside the Engine

What happens on every request

Five deterministic steps. No black box. Every answer is traceable.

1 📨
Intercept + PII Scan
STOP/HELP keywords handled for TCPA compliance. PII regex detects SSNs, card numbers, addresses — redacted before the message is logged or processed.
STOP_WORDS / HELP_WORDS + pii_redact()
▶
2 🌎
Detect Language
Spanish keyword heuristic checks for indicators like "necesito", "ayuda", "impuestos".
detect_language() -> "en" | "es"
▶
3 🎯
Two-Tier Intent Matching
Tier 1: Keyword scorer against 20 curated intents. Sub-50ms, zero external calls. Handles 70-80% of traffic.

Tier 2: Low confidence? Falls back to Claude Sonnet for classification. 8-second circuit breaker timeout — if the LLM is slow or down, the keyword path still works.
IntentMatcher.match() → keyword_score() || llm_classify() → (intent, confidence)
▶
4 📋
Format for Channel
SMS gets short text with numbered steps. Web gets rich JSON with sources and deadlines.
format_sms() | format_web()
▶
5 🛡️
Safety Validation
Every phone number and URL is checked against the approved list. No hallucinated contacts.
validate_response_citations()

Architectural Decision

"Why didn't you use RAG?"

20 intents don't need vector search. This is a deliberate choice for reliability.

Typical RAG Chatbot
  • ✗ Embedding pipeline + vector DB
  • ✗ Chunk management + similarity tuning
  • ✗ Every request hits an LLM — latency + cost
  • ✗ Hallucination risk on every response
  • ✗ Fails completely if LLM provider goes down
Hey804 Engine
  • ✓ Keyword scorer handles 70-80% of traffic
  • ✓ Sub-50ms, zero external calls on keyword path
  • ✓ LLM only used for classification, not generation
  • ✓ Every answer from verified static JSON corpus
  • ✓ If LLM goes down, keyword path still serves residents

The right amount of AI is the minimum needed to be reliable. Not the maximum available.

Production Path

How this plugs into the city

Hey804 doesn't replace the city's existing systems. It becomes a new delivery channel that plugs into them.

Emergency Broadcasts
In production, POST /api/broadcast becomes a webhook receiver behind the city's existing emergency management system (Everbridge / Richmond Ready Alerts). The city's approval workflows, SSO, and two-person confirmation stay in place. Hey804 is just another endpoint that fires when they send an alert — like their email system, but for SMS + web + widget.
Widget Partner Analytics
The widget embeds with <script src="widget.js?partner=salvation-army">. The partner param flows through the API as context. The city can report back: "Your visitors asked about rent help 40 times this month." Per-partner analytics without the partner building anything.
Data Layer
SQLite with WAL mode handles three concerns: subscriber state (who texted, language preference, opted-out status), conversation logging (every Q&A pair with channel + intent + timestamp for usage analytics), and alert persistence (active alerts that web/widget clients poll). Zero infrastructure cost. Swap for Postgres in production without changing a line of application code.
